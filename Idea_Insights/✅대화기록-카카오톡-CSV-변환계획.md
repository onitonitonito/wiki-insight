### 📋 1. 통합 CSV 변환 계획 (Action Plan)

**목표:** 두 파일의 텍스트 데이터를 파싱하여 `Date(날짜)`, `Time(시간)`, `Sender(발신자)`, `Message(메시지)`의 4개 컬럼을 가진 구조화된 통합 CSV로 변환.

1. **정규표현식(Regular Expression) 이원화 설계:**
   각 기기별 카카오톡 메시지 발생 패턴의 **첫 줄**을 정확히 잡아내는 두 개의 정규식을 준비합니다.
   - **Android 패턴 (기준):** `^(\d{4})년 (\d{1,2})월 (\d{1,2})일 (오전|오후) (\d{1,2}):(\d{2}), (.*?) : (.*)`
   - **iOS 패턴:** `^(\d{4})\. (\d{1,2})\. (\d{1,2})\. (오전|오후) (\d{1,2}):(\d{2}), (.*?) : (.*)`
2. **순차적 줄 단위 파싱 (Line-by-Line Parsing):**
   - 텍스트 파일을 한 줄씩 읽어 내려가며 위 두 정규식 중 하나에 매칭되는지 검사합니다.
   - 매칭될 경우: 기존에 누적하던 메시지 버퍼를 닫고 하나의 행(Row)으로 저장한 뒤, 새로운 `Date`, `Time`, `Sender` 변수를 할당하고 `Message` 버퍼를 새로 시작합니다.
   - 매칭되지 않을 경우: 현재 활성화된 `Message` 버퍼에 개행 문자(`\n`)와 함께 해당 줄의 텍스트를 이어서 붙입니다. (다중 줄 작업일보 보존)
3. **데이터 표준화 (Standardization):**
   - 추출된 날짜와 시간을 안드로이드 기반 또는 글로벌 표준(예: `YYYY-MM-DD`, `HH:MM` 24시간제)으로 데이터 타입을 통일 변환합니다.
4. **CSV 내보내기:**
   - 파싱된 데이터를 Pandas DataFrame 등으로 구조화한 후, 본문 내의 쉼표(,)나 개행문자(\n)로 인해 CSV 열이 깨지지 않도록 모든 텍스트 필드를 쌍따옴표(`""`)로 감싸는 이스케이프(Escape) 처리를 적용하여 저장합니다.

### 🚨 2. 예상 문제점 및 🛠️ 해결 방안

#### 문제점 1. 불필요한 시스템 텍스트 및 헤더 노이즈 
115 파일(안드로이드)의 최상단에는 웹 클립보드 복사 흔적(Gemini 관련 UI 텍스트)이 섞여 있고, 중간중간 `2026년 1월 22일 목요일`과 같은 단순 날짜 변경선이 존재합니다. 이를 본문이나 발신자로 오인하면 데이터가 어긋납니다.
- **해결 방안:** 
  첫 번째 정규식 매칭(실제 대화 시작)이 일어나기 전까지 나타나는 모든 정규식 미매칭 텍스트 라인은 **무조건 무시(Skip)** 하도록 로직을 짭니다. 또한, 카카오톡의 날짜 변경선은 특정 정규식(예: `^\d{4}년 \d{1,2}월 \d{1,2}일 요일$`)으로 필터링하여 메시지 본문에 섞이지 않도록 예외 처리합니다.

#### 문제점 2. 방대한 다중 줄(Multi-line) 메시지에 의한 CSV 열 깨짐 현상
현장 작업 일보의 특성상 하나의 메시지가 수십 줄로 작성되어 있습니다. 단순한 구분자(,) 처리 시 엑셀 등에서 CSV를 열 때 줄바꿈 때문에 행이 산산조각 날 위험이 큽니다.
- **해결 방안:**
  파이썬의 `csv` 라이브러리 내장 옵션인 `quoting=csv.QUOTE_MINIMAL` 또는 `QUOTE_ALL` 옵션을 적용합니다. 이렇게 하면 내부에 줄바꿈(`\n`)이 들어간 메시지 전체가 하나의 `"텍스트 뭉치"`로 인식되어 CSV의 단일 셀(Cell) 안에 안전하게 안착합니다.

#### 문제점 3. 메시지 본문 내 정규식과 유사한 패턴의 등장 (오작동 위험)
누군가가 이전 메시지를 복사해서 본문에 붙여넣었을 경우 (예: 본문 안에 `2025. 5. 17. 오전 7:29, 유종수 : ...` 가 텍스트로 들어있는 경우), 스크립트가 이를 새로운 메시지의 시작으로 착각해 메시지를 분단시킬 수 있습니다.
- **해결 방안:**
  정규식 패턴 설계 시 **문자열의 맨 처음을 의미하는 `^` 앵커(Anchor)** 를 반드시 사용합니다. `^2025...` 형태로만 매칭되게 하면, 띄어쓰기나 텍스트 중간에 섞인 유사 패턴에 의한 파싱 오류를 원천 차단할 수 있습니다.

#### 문제점 4. 연도 및 시제 불일치 (2025년 vs 2026년)
iOS 파일은 2025년으로 표기되어 있고, 안드로이드 파일은 2026년으로 표기되어 시계열의 차이가 존재합니다. 
- **해결 방안:** 
  현재는 두 파일의 텍스트에 표기된 연도(YYYY)를 원형 그대로 추출하는 데 집중합니다. 파싱이 모두 완료되어 단일 통합 DataFrame이 만들어진 후, 날짜 열(`Date`)을 기준으로 **오름차순 정렬(Sort)** 을 수행하여 데이터베이스의 시간적 일관성을 확보합니다.
