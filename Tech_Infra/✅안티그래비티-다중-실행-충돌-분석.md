> [!IMPORTANT]
> **핵심 Takeaway: 다중 에이전트 동시 실행 리스크**
> 동일한 작업 공간(폴더)에서 Antigravity IDE와 다수의 CLI를 동시 실행하는 것은 **파일 시스템 경합(Race Condition)과 컨텍스트 불일치(Context Desynchronization)**를 유발하여 치명적인 코드 손실이나 논리적 충돌을 초래할 수 있습니다. 다중 에이전트 작업 시에는 철저한 역할 분담(Role Delegation)과 작업 영역(Target File/Path)의 명확한 분리가 필수적입니다.

## 1. 개요
한 프로젝트 폴더 내에서 시각적 도구인 Antigravity IDE 1개와 터미널 기반의 Antigravity CLI 2개 이상을 동시에 띄워놓고 작업하는 환경(Multi-Agent / Multi-Instance)은 겉보기에는 생산성을 극대화할 수 있을 것 같지만, 아키텍처 상 심각한 병목과 충돌을 야기합니다.

## 2. 핵심 문제점 분석 (1 IDE + N CLI 동시 작업 시)

### 2.1 파일 시스템 충돌 및 덮어쓰기 (Race Conditions)
- **동시 수정 이슈**: IDE에서 특정 파일(`app.js`)을 수정하는 동안, CLI 1이 동일한 파일에 접근하여 코드를 삽입하거나 포맷팅을 수행할 수 있습니다. 두 에이전트가 동시에 파일을 편집하려고 하면 마지막에 저장한 에이전트의 작업만 남고 이전 작업은 덮어쓰기(Overwrite)되어 데이터가 유실됩니다.
- **파일 잠금(File Lock)**: Windows 환경 등의 경우 한 프로세스가 파일을 점유하고 있을 때 다른 프로세스가 접근하면 접근 거부(Permission Denied) 에러가 발생하여 에이전트의 워크플로우가 중단될 수 있습니다.

### 2.2 컨텍스트 불일치와 환각 (Context Desynchronization)
- **독립된 대화 기록(Conversation History)**: IDE와 CLI 1, CLI 2는 각각 고유한 대화 컨텍스트를 가집니다. 즉, CLI 1이 방금 생성하거나 리팩토링한 파일의 최신 상태를 IDE나 CLI 2는 즉각적으로 인지하지 못합니다.
- **환각(Hallucination) 증가**: IDE는 과거의 파일 상태(캐시된 컨텍스트)를 바탕으로 다음 작업을 계획하지만, 실제 파일 시스템은 이미 CLI에 의해 변경된 상태일 수 있습니다. 이로 인해 존재하지 않는 라인(Line)을 수정하려고 시도하거나, 이미 삭제된 변수를 참조하는 코드를 작성하는 등 심각한 환각 오류가 발생합니다.

### 2.3 형상 관리(Git) 충돌 및 락(Lock) 에러
- **Git Index Lock**: 여러 에이전트가 동시에 `git add`나 `git commit`을 시도하면 `.git/index.lock` 파일 충돌이 발생하여 형상 관리가 꼬이게 됩니다.
- **의존성 설치 충돌**: 한 CLI가 패키지(예: `npm install`)를 설치하는 도중 다른 CLI가 모듈을 추가하거나 빌드를 돌리면 패키지 락 파일(`package-lock.json`) 등이 손상될 수 있습니다.

## 3. 다중 모델 사용 시나리오별 분석

다수의 CLI와 IDE 인스턴스에 할당된 AI 모델이 같을 때와 다를 때, 발생하는 문제의 양상이 다릅니다.

### 3.1 모든 인스턴스가 동일한 모델을 사용하는 경우 (예: 모두 고성능 모델)
- **API 토큰 및 Rate Limit 고갈**: 동일한 모델의 API 쿼터를 동시에 소모하므로, 단일 모델에 대한 분당 요청 수(RPM) 또는 분당 토큰 수(TPM) 제한에 급격히 도달합니다. 한 에이전트가 긴 코드를 생성하는 동안 다른 에이전트들은 `429 Too Many Requests` 등의 에러로 중단될 확률이 높습니다.
- **일관된 시각, 중복된 노력**: 모델의 추론 능력(Capability)이 동일하므로, 복잡한 문제를 각자 동일한 방식으로 풀려고 시도하여 불필요하게 동일한 파일을 건드리는 병목 현상이 심화될 수 있습니다.

### 3.2 각 인스턴스가 다른 모델을 사용하는 경우 (예: IDE=고성능, CLI=경량 모델)
- **코드 품질 및 아키텍처 불일치 (Quality Mismatch)**: 
  - **IDE (고지능 모델)**: 프로젝트 전체 아키텍처와 디자인 패턴을 고려하여 추상화된 고급 코드를 작성합니다.
  - **CLI (경량/저지능 모델)**: 작업 속도는 빠르지만 시야가 좁아, IDE가 짜놓은 구조를 무시하고 임시방편(Hardcoding) 코드나 스파게티 코드를 강제로 주입할 수 있습니다.
- **의도 충돌 (Intent Clash)**: 고성능 모델은 리팩토링을 통해 컴포넌트를 분리하려 하고, 저성능 모델은 단일 파일에 기능만 빠르게 덧붙이려 하다 보니 코드베이스가 혼란스러워집니다.

## 4. 해결책 및 Best Practices (멀티 에이전트 협업 가이드)

동일 폴더에서 여러 에이전트를 효과적으로 쓰려면 아래의 규칙을 준수해야 합니다.

1. **상호 배타적 영역 분리 (Domain Isolation)**: 
   - IDE: `src/components/` (프론트엔드 담당)
   - CLI 1: `server/` (백엔드 API 담당)
   - CLI 2: `tests/` (테스트 코드 작성 담당)
   - 이처럼 각 에이전트가 다루는 디렉토리와 관심사를 철저히 분리해야 파일 덮어쓰기를 막을 수 있습니다.
2. **동기화 포인트 설정 (Sync Protocol)**: 특정 에이전트의 큰 작업이 끝나면, 다른 에이전트들에게 "디렉토리를 다시 읽어라(Read directory/Refresh context)"라고 명시적으로 지시하여 컨텍스트를 동기화해야 합니다.
3. **Subagent 아키텍처 활용 권장**: 개별적으로 여러 인스턴스를 띄우기보다는, 주 에이전트 내에서 **`invoke_subagent`** 기능 등을 활용하여 하위 에이전트(Subagent)를 파생시키는 것이 상태 동기화 및 작업 분배 측면에서 훨씬 안전합니다.

## 5. 멀티 CLI 및 에이전트 다중 작업의 글로벌 사례 및 웹 리서치

최신 AI 코딩 에이전트(Claude Code, Devin, Aider 등) 커뮤니티와 개발자 생태계의 웹 리서치 결과, 동일 폴더 내 다중 인스턴스 실행은 **'단순 병렬 처리(Naive Parallelism)'의 실패 사례**로 규정되며, 이를 극복하기 위한 다양한 접근법이 시도되고 있습니다.

### 5.1 CLI 복수 개 실행 시 발생하는 글로벌 유저 후기 및 이슈
- **상태 파일 충돌 (State Coordination Collisions)**: Aider나 Claude Code 같은 도구는 작업 이력을 로컬 폴더(예: `.aider` 또는 `.gemini`)에 기록합니다. 두 개 이상의 CLI를 같은 폴더에서 띄우면, 이 숨김 파일들에 동시 쓰기(Write)가 발생하여 에이전트의 로컬 데이터베이스가 손상되는 사례가 Github 이슈에 다수 보고되었습니다.
- **연쇄적 코드 파괴 (Cascading Breakage)**: 한 개발자의 레딧(Reddit) 후기에 따르면, 에이전트 A가 API 구조를 변경하여 테스트를 통과시켰으나, 에이전트 B가 동시에 구형 API 구조를 기반으로 프론트엔드 라우터를 작성해버리는 바람에 "아무 에러도 안 났는데 전체 빌드가 깨지는" 치명적인 조용한 실패(Silent Failure)를 겪었습니다.
- **결론**: 커뮤니티에서는 아무런 격리 조치 없이 터미널 여러 개에 CLI를 띄우는 행위를 **"레포지토리의 혼돈(Recipe for Chaos)"**이라 부르며 강력히 경고하고 있습니다.

### 5.2 AGENT 프레임워크를 활용한 멀티 에이전트 운용 사례
글로벌 기술 커뮤니티에서는 이러한 충돌을 막기 위해 **"조정 레이어(Coordination Layer)"**를 둔 AGENT 프레임워크 접근 방식을 채택하고 있습니다.
- **역할 및 상태 공유 (STORM, ccmanager 등)**: 연구 목적으로 개발된 프레임워크나 기업용 관리 도구들은 `TURNFILE` 또는 `WORKLOG` 같은 상태 파일을 리포지토리 루트에 두고, 어떤 에이전트가 무슨 파일을 점유하고 있는지 서로 공유(Locking)합니다.
- **역할의 세분화 (Human-in-the-loop)**: 에이전트를 Architect, Implementer, Reviewer 등으로 나누어, Architect가 먼저 `PRD.md`나 `VISION.md`를 작성하면 나머지 하위 에이전트(Subagents)들이 이를 읽고 충돌하지 않는 각자의 디렉토리에서만 작업하도록 유도합니다.

### 5.3 업계 표준 해결책: Git Worktrees (작업 트리 분리)
다수의 AI 에이전트를 동일한 코드베이스에서 가장 안전하게 돌리는 업계 표준(Consensus Standard)은 **Git Worktrees**를 사용하는 것입니다.
- **개념**: 단일 Git 리포지토리를 참조하되, 물리적인 작업 폴더를 여러 개로 분리(`git worktree add ../agent-task-1 -b feature/A`)하는 방식입니다.
- **효과**: CLI 1번과 CLI 2번이 완전히 분리된 폴더 경로에서 실행되므로, 파일 덮어쓰기나 인덱스 락(`index.lock`) 에러가 원천 차단됩니다. 각 에이전트는 독립적인 환경에서 원자적 커밋(Atomic Commit)을 수행할 수 있으며, 최종적으로 개발자(Human)가 PR을 통해 브랜치를 안전하게 병합합니다.
